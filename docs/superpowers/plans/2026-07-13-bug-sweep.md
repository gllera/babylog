## Part A — the 5 confirmed bug fixes

### Fix 1 (HIGH) — `get_stats` NULL diaper counts → `src/tools.ts:1534-1536`
Over a window with **zero** diaper rows, `SUM(CASE …)` returns `NULL` (SQLite), which
lands in `structuredContent.diapers.pee_count/poop_count` — declared non-nullable
`z.number().int()` (`tools.ts:1478-1479`). The MCP SDK's `validateToolOutput`
(`@modelcontextprotocol/sdk/.../mcp.js:200-206`) then throws
`McpError(InvalidParams, "Output validation error…")`, so the whole tool call fails.
Trivially reachable: fresh baby, narrow `since`/`until`, or `window:"today"` before the
first diaper.

Wrap both sums in `COALESCE(…, 0)`:

```sql
COALESCE(SUM(CASE WHEN kind = 'pee'  THEN 1 ELSE 0 END), 0) AS pee_count,
COALESCE(SUM(CASE WHEN kind = 'poop' THEN 1 ELSE 0 END), 0) AS poop_count,
```

- SQL-level = single source of truth; mirrors the feeding aggregate two lines up; the
  read-site cast and `structuredContent` need no change. (The Alexa path already does
  `?? 0` at `alexa.ts:729-730`.)
- **Verify:** `tsc`; manual `get_stats` on seeded baby #1 with no diapers → `0/0`, no
  validation error.

### Fix 2 (MEDIUM) — `check_indications` accepts impossible dates → `src/tools.ts:~1266`
`date` is validated by shape only (`.regex(/^\d{4}-\d{2}-\d{2}$/)`), so `"2026-02-30"`
passes, then `madridDayWindow → addDaysIso → new Date(...).toISOString()` throws
`RangeError` (`lib.ts:188`). Add the same real-date guard the baby tools use
(`isValidIsoDate` is already imported in tools.ts):

```ts
date: z.string()
  .regex(/^\d{4}-\d{2}-\d{2}$/)
  .refine(isValidIsoDate, "must be a real ISO date (YYYY-MM-DD)")
  .optional()
  .describe(...)
```

- **Verify:** `tsc`; manual `check_indications({date:"2026-02-30"})` → clean validation
  error instead of a thrown `RangeError`.

### Fix 3 (LOW) — Alexa "wet+dirty" misread as pee → `src/alexa.ts`
The combined-diaper regex (line 601) lists pee/poop words but omits `wet`/`dirty`/
`mojado`/`sucio` (present in the single-kind branches), so `"wet and dirty"` falls
through to the pee branch — the opposite of the documented rule. Two parts:

1. **Extract** the raw→kind logic (lines ~596-616) into a pure exported helper and call
   it from `handleRecordDiaper`:
   ```ts
   export function classifyDiaperKind(raw: string): DiaperKind | null { … }
   ```
2. **Fix the "both" regex** — add the wet/dirty words in both orders:
   `(pis|pip[ií]|pee|wee|mojado|wet)` … `(caca|pop[oó]|poop|poo|sucio|dirty)`.

- **Verify:** add cases to `test/alexa.test.ts` — `classifyDiaperKind("wet and dirty")
  === "poop"`, `"pee and dirty" === "poop"`, plus regressions `"pee"`, `"caca"`,
  `"pis y caca" === "poop"`. `npx vitest run`.

### Fix 4 (LOW) — dashboard `est_height_cm` anchored to `now` → `src/api.ts:522-526`
Project to `new Date(dayStart)` instead of `now`, matching `est_weight_g` and the MCP
path (the divergence the surrounding comments warn about):

```ts
est_height_cm: estimateHeightCm(recentHeights.results as …, new Date(dayStart), sel.baby.date_of_birth),
```

- Field is currently unused by the client → zero-risk consistency change. **Verify:** `tsc`.

### Fix 5 (LOW) — `markWhen` yesterday-tooltip off-by-one → `src/app.html:3008`
Cutoff `(RS_DAYS - 1) * 1440` → `(RS_DAYS - 2) * 1440`, so yesterday's tape-mark
tooltips show time-only, agreeing with the function's comment and with
`markerAbsLabel`/`stripRowWhen`.

- **Verify:** reasoning (no JS test harness); optional manual — hover a yesterday mark,
  expect `"22:05"` not `"15/07/2026 22:05"`.

## Part B — 2 optional hardening notes (decision needed)

- **B1 — merged-feeding undo clamp** (`src/app.html:2143`, `savedToast`): when a merge
  was clamped at `MAX_FEEDING_ML` (5000), the subtract-undo restores a slightly wrong
  original. Fold in a guard: skip the subtract-undo when
  `created.amount_ml === MAX_FEEDING_ML`, falling back to a no-undo toast. Trivial/safe
  — **recommend include**.
- **B2 — Alexa cert chain not validated to a trusted root** (`src/alexa.ts:387-411`):
  a real change, not a one-liner — proper chain-to-root validation in workerd, with a
  live risk of breaking working production Alexa auth. Already mitigated by the
  `s3.amazonaws.com/echo.api/` path lock + `applicationId` check. **Recommend defer** to
  its own branch/PR with isolated testing.
